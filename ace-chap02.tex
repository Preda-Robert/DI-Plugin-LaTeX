\section{Purpose and responsibilities}
\label{chap2:purpose}

The plugin is structured as a \textbf{thin VS Code host} (\texttt{extension.ts}) orchestrating analysis, workspace aggregation, user commands, and editor integrations, plus \textbf{specialized modules} for reusable algorithms. Table~\ref{tab:modules} summarizes module responsibilities.

\begin{table}[htbp]
\centering
\caption{Software modules and responsibilities}
\label{tab:modules}
\begin{tabular}{|l|p{9.5cm}|}
\hline
\textbf{Module} & \textbf{Responsibilities} \\
\hline
\texttt{src/csharp/analyzer.ts} & Parse C\# with tree-sitter; extract constructors; detect file-local DI issues. \\
\hline
\texttt{src/extension.ts} & Activate extension; workspace summary; commands; diagnostics; code actions; code generation. \\
\hline
\texttt{src/registrationInsert.ts} & Locate composition-root insertion points; merge \texttt{services.Add*} lines; parse generated \texttt{AppBuilder.cs}. \\
\hline
\texttt{src/extractInterface.ts} & Build workspace edits for ``extract interface and register'' quick fixes. \\
\hline
\texttt{package.json} & Command contributions, activation events, dependency manifest. \\
\hline
\end{tabular}
\end{table}

At runtime, all analysis flows \emph{from text buffers} (open documents and workspace files) to \emph{structured summaries} (\texttt{WorkspaceDiSummary}) and then to \emph{user-visible artifacts} (Output channel text, diagnostics, \texttt{WorkspaceEdit}, files under \texttt{GeneratedDI/}).

\section{Key data structures (cross-cutting)}
\label{chap2:data-structures}

The host defines \texttt{WorkspaceDiSummary}, the central aggregate built by \texttt{buildWorkspaceSummary()}:
\begin{itemize}
	\item \texttt{files} --- opened C\# documents with source text;
	\item \texttt{constructors} --- \texttt{ConstructorInfo} enriched with \texttt{fileName} and \texttt{relativePath};
	\item \texttt{registeredTypes} --- types inferred from \texttt{AddSingleton|Scoped|Transient}, \texttt{AddDbContext}, \texttt{AddAutoMapper}, Identity APIs, etc.;
	\item \texttt{interfaceToImpl} --- map from interface name to implementing class names (regex on \texttt{class X : IY});
	\item \texttt{typeToNamespace} --- simple name to namespace for generated usings;
	\item \texttt{hasCompositionRoot}, \texttt{aspNetMvcHost}, \texttt{genericClassNames}, \texttt{nonInjectableClassNames} --- feature flags and exclusion sets.
\end{itemize}

The analyzer exports \texttt{CSharpDiAnalysis}, the per-file result consumed by diagnostics and file-scoped commands. Detailed field definitions appear in Chapter~\ref{chap3:analyzer}.

\section{Technology stack and integration points}
\label{chap2:tech}

\begin{itemize}
	\item \textbf{Runtime:} Node.js; TypeScript compiled to \texttt{out/extension.js}.
	\item \textbf{Parsing:} \texttt{tree-sitter} native binding + \texttt{tree-sitter-c-sharp} grammar~\cite{tree-sitter,tree-sitter-csharp}.
	\item \textbf{Editor API:} VS Code Extension API~\cite{vscode-api} --- \texttt{commands}, \texttt{languages.createDiagnosticCollection}, \texttt{registerCodeActionsProvider}, \texttt{workspace.applyEdit}, \texttt{workspace.fs}.
	\item \textbf{Target DI API:} \texttt{Microsoft.Extensions.DependencyInjection} registration idioms only.
\end{itemize}

Activation occurs on \texttt{onLanguage:csharp}. Subscriptions include document open/change/close handlers that refresh diagnostics for C\# buffers.

\section{Design principles}
\label{chap2:design}

\begin{itemize}
	\item \textbf{Fail soft on parse errors.} Large files and native parse failures return empty constructor lists with warning messages instead of crashing the extension host.
	\item \textbf{Separate registration from factory concerns.} Suggest/generate commands for factories output \texttt{*Factory.cs} snippets only; \texttt{services.Add*} lines are produced by registration commands (Chapter~\ref{chap4:extension}).
	\item \textbf{Conservative factory rule.} Factories are generated only when a constructor has at least three parameters and each parameter type is a known interface key in \texttt{interfaceToImpl}.
	\item \textbf{Composition-root-first insertion.} Automated edits target \texttt{CompositionRoot}, \texttt{Add*Services} extension methods, or \texttt{ConfigureServices}, not arbitrary files.
\end{itemize}

\section{Known limitations (system level)}
\label{chap2:limitations}

\begin{itemize}
	\item No Roslyn semantic model: generic constraints, \texttt{using} aliases, and partial classes across files may be misclassified.
	\item Workspace scan capped at 200 \texttt{.cs} files per operation.
	\item Heuristic registration detection cannot model every third-party DI extension method.
	\item Generated code is a starting point; lifetimes and architectural boundaries remain the developer's responsibility.
\end{itemize}
